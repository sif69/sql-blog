Which database should you use? A practical guide for developers

The decision between relational and non-relational databases isn't just technical — it shapes how your application scales, performs, and evolves. Here's what you actually need to know, with real-world examples to make it concrete.

DB

Database Engineering

8 min read · System Design Series

For most developers, a relational database like MySQL or PostgreSQL is the right default. They've been around for 40+ years and solve the vast majority of problems elegantly. But when your use case pushes against their limits, you need to know when and why to reach for a non-relational (NoSQL) database instead.

Below are four scenarios where NoSQL databases genuinely shine — backed by real-world examples from companies you already know.

When NoSQL makes sense — 4 real scenarios

⚡
Scenario 1

Your application requires super-low latency

When milliseconds cost you millions

Traditional relational databases hit a wall when you need responses in under a millisecond. Disk I/O, complex joins, and ACID transaction overhead simply can't keep up. In-memory NoSQL databases like Redis store data directly in RAM, cutting response times to microseconds.

Real-world example — Twitter / X's timeline

When you open Twitter, your home timeline loads almost instantly — even with hundreds of millions of users active at the same time. Twitter uses Redis to cache pre-computed timelines in memory. Instead of running a massive SQL JOIN across followers, tweets, and users every time you refresh, Redis returns the pre-built timeline in under a millisecond. The database query already happened — Redis just hands you the result.

Scenario 2

Your data is unstructured or has no fixed schema

When every record looks a little different

Relational databases demand a rigid schema — every row must fit the same columns. When your data varies wildly from record to record (different fields, nested objects, optional attributes), that rigidity becomes painful. Document databases like MongoDB store each record as a flexible JSON document, so no two records need to look the same.

Real-world example — an e-commerce product catalog

Amazon sells everything from books to refrigerators. A book has an ISBN, author, and page count. A refrigerator has voltage, capacity, and dimensions. A t-shirt has size and color variants. In a relational database, you'd need dozens of nullable columns or a complex EAV (entity-attribute-value) pattern. With MongoDB, each product is just a document with whatever fields make sense for that item — no empty columns, no awkward workarounds.

SQL approach — painful

One huge products table with 50+ columns, most of them NULL for any given row. Adding a new product type means a database migration.

MongoDB approach — flexible

Each product is a JSON document. Books get book fields, fridges get fridge fields. New product types need zero schema changes.

Scenario 3

You only need to serialize and deserialize data

When you're just storing and retrieving blobs of data

Some data doesn't need relationships at all. You write it once, read it back as-is, and never join it with anything else. Think user session data, configuration blobs, API response caches, or feature flags. For this, the relational model adds complexity with zero benefit — a key-value store is all you need.

Real-world example — Netflix user sessions and preferences

When you log into Netflix, the platform loads your watch history, preferred language, subtitle settings, and UI preferences almost instantly. Netflix stores all of this as serialized JSON in Amazon DynamoDB. There's no need to join across multiple tables — your session data is simply written as a blob when you log in and read back when you return. DynamoDB's key-value model is a perfect fit: look up by user ID, get back everything about that session in one shot.

"We use DynamoDB for use cases that require high-throughput, low-latency reads and writes, such as storing and serving hundreds of billions of items."

— Netflix Tech Blog

Other examples include storing JWT tokens, caching API gateway responses, saving shopping cart contents temporarily, or holding webhook payloads before they're processed.

Scenario 4

You need to store a massive amount of data

When your data outgrows a single machine

Relational databases scale vertically — you buy a bigger server. That works up to a point, then it gets prohibitively expensive and hits hardware limits. Column-family stores like Apache Cassandra and HBase are built to scale horizontally — just add more nodes. They're designed from the ground up to handle petabytes of data spread across thousands of machines.

Real-world example — Apple's iCloud and Cassandra

Apple uses Apache Cassandra to handle the metadata and activity data for iCloud — covering over a billion Apple devices. Every time you back up a photo, sync a note, or send an iMessage, that activity is logged. The volume is staggering: Apple runs one of the largest known Cassandra deployments in the world, managing over 75,000 Cassandra nodes globally. No single relational database server — no matter how powerful — could handle this. Cassandra's distributed architecture spreads the load across thousands of machines automatically.

Still not sure which to pick for your use case? The honest answer is: start relational. Only move to NoSQL when you hit a concrete problem that SQL can't solve elegantly.
